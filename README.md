# NomNom ‚Äî Food Ordering Prototype

PHP/CSS/jQuery prototype for the **CBSC4103 Software Construction** assignment. Demonstrates the customer-facing flow before wiring the real PHP backend logic.

---

## Quick start

The prototype is now a **PHP application**. You will need a PHP server (like Docker, XAMPP, or PHP's built-in server) to run it. The **service worker requires HTTPS or `localhost`** to register, so for the full PWA experience use a local web server.

### Run on a local server (full PWA)
From the `prototype/` folder, run:

```powershell
# PHP built-in server
php -S localhost:8000
```
Or use the provided `docker-compose.yml` to spin up the full Apache/PHP and MariaDB stack.

Then open http://localhost:8000

## What's wired vs. stubbed

### Wired (real interactivity, working in browser)
- Add-to-cart on restaurant detail page ‚Üí updates cart badge & localStorage
- Cart page: qty +/‚àí, remove, clear, live totals
- Checkout: delivery method toggle adjusts delivery fee in summary
- Service worker registration + offline fallback page
- PWA install prompt (browser-controlled)
- Toast notifications

### Stubbed (placeholder for real PHP logic)
- Form submissions (login, register, checkout) ‚Üí simulate success then redirect
- CSRF token inputs ‚Üí `DUMMY_CSRF_TOKEN_REPLACE_SERVER_SIDE` placeholder
- Push notification opt-in ‚Üí toast only (real version calls Push API)
- Promo code application ‚Üí toast only
- All restaurant cards link to the same `restaurant-detail.php`

## Known limitations of the prototype

1. No real backend logic yet ‚Äî all "saves" are toasts.
2. Cart is per-browser via `localStorage`, not shared across devices.
3. Restaurant cards all link to the same detail page (intentional ‚Äî it's one design template).
4. No real payment integration (Stripe Elements / Billplz iframe would replace the static card form).
5. No real-time order tracking ‚Äî the 4-step tracker is a static visual mock.
